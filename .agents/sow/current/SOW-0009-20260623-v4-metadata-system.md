## Requirements

### Purpose

Make a v4 file **self-describing** by embedding per-scope metadata, so a v4 `.iprdb` is a
complete, portable threat-intel artifact (data + metadata) — the project's stated
direction. Multi-feed files are the norm, so metadata is **per-scope**, with the three
universal, hot fields (`name`, `version`, `type`) seekable directly (no KV scan / no KV
rewrite) and everything else in an opaque per-scope key-value store.

### User Request / Decisions (2026-06-23)

1. **Metadata is per-scope** (multi-feed is the norm); the file header stays purely
   structural.
2. **Three seekable per-scope header fields** — `name`, `version`, `type` — must be
   *seek to get, seek to set* (no KV search, no full-KV rewrite).
3. **`version`** is caller-bumped via standard helpers (`scopeBumpVersion` /
   `scopeSetVersion`), **not** engine-auto (the engine can't know which feeds a data
   mutation touched without decoding opaque scope).
4. **Per-scope KV** for everything else: `key` (UTF-8 ≤1024, no NUL, non-empty),
   `value` (opaque bytes, **any size, no limit**), `type` (**u32**: 0=text validated,
   non-zero = caller-defined binary). **Unbounded** in entry count and value size.
5. KV on-disk = a **bulk-loaded B+tree** behind `kv_root` (overflow page chains for large
   values), **full-rewrite per scope per commit** (buffer in memory → bulk-rebuild at
   commit; no incremental split/merge), `O(log n)` descent reads. CoW: fresh pages + free
   old; the derived allocator walks it.
6. **Metadata reads belong on the `Reader`** (decision 2026-06-24). The initial
   implementation placed the metadata *read* API (`scope_list`/`scope_name`/`scope_version`/
   `scope_type`, `meta_get`/`meta_list`) only on the `Writer`, which holds `LOCK_EX`
   (`os.rs:155,172`) — so a read-only consumer could not read a feed's metadata without an
   exclusive lock, defeating the self-describing purpose for the concurrent shared-lock read
   path (`MmapReader`/`LOCK_SH`, `os.rs:56-80`). **Resolution:** expose the same metadata
   read API on the **`Reader`** in both languages, descending the **on-disk committed tree**
   (already validated at open — `reader.rs` scope+KV validate), mirroring how SOW-0008 placed
   the cursor on the `Reader`. The `Writer`-side reads stay (they serve the in-memory registry
   during a txn). This slightly expands SOW-0009 scope; conformance reads metadata via the
   `Reader`.

### Assistant Understanding

The exact layout, semantics, and CoW model are in `design-iprange-v4-scope-api.md`
§C.1–C.6. Summary of the on-disk additions:
- meta: `@90 scope_table_root u32` → `meta_size` 94, `version_minor` 1.
- **Scope table** — a **new (separate) fixed-record B+tree** keyed by `scope_id` (own value
  layout, not a reuse of the IP tree); entry =
  `{scope_id u32, version u64, type u8, name_len u16, name u8[256], kv_root u32}`.
- **Per-scope KV** — a **bulk-loaded B+tree** behind `kv_root` (unbounded; overflow pages for
  large values), full-rewrite-on-change, `O(log n)` reads.
- **File/dataset metadata** = reserved `scope_id 0` (the `FILE` target).
- Forward-compat: v4.0 readers read the IP tree and skip metadata; v4.0 writers MUST
  refuse a v4.1 file.

## Pre-Implementation Gate

### Problem / root-cause model

A v4.0 file carries IP data but no metadata, so it is not a self-describing distributable
artifact and a consumer needs a side channel for feed identity/attributes. Embedding
per-scope metadata (with the hot fields seekable) makes the file complete without bloating
the per-feed access paths.

### Evidence reviewed

- `design-iprange-v4-livedb.md` (locked core: meta layout §5.1, forward-compat rules,
  COW/allocator/commit), `design-iprange-v4-scope-api.md` (§C — the full design),
  `update-ipsets-v4-adoption-findings.md` (consumers carry ~80 metadata fields/feed).

### Affected contracts and surfaces

- **Format change:** meta gains `scope_table_root`; two new on-disk structures (scope
  table; per-scope KV runs); `version_major` unchanged, `version_minor` 0 → 1.
- **New API:** scope-table access (`scopeDefine`/`scopeDrop`/`scopeName`, the per-scope
  header get/set, `scopeBumpVersion`/`scopeSetVersion`), per-scope KV
  (`metaSet`/`metaGet`/`metaDelete`/`metaList`).
- **Depends on:** SOW-0008 cursor/helpers (for ordered access and the mutate-during-open
  contract used by KV reads/writes).

### Existing patterns to reuse

The locked v4 core: the fixed-record B+tree *pattern* (the scope table is a **new** tree
with its own value layout, not a code reuse of the IP tree), COW + dirty-set + double-meta
commit, the derived allocator (extended to walk `scope_table_root` + each `kv_root`'s KV
tree + overflow pages), per-page CRC, the bounds-safety/fuzz discipline from SOW-0007, and
the v3/v4 shared conformance harness.

### Risk and blast radius

- **Format-version step** (v4.1): the forward-compat behavior (old reader skips, old
  writer refuses) must be implemented and tested precisely, or a v4.0 writer could free
  metadata pages.
- **KV validation** (UTF-8/no-NUL on keys and `type==0` values) must be cross-language
  identical.
- **Full-rewrite KV** must buffer in memory and bulk-rebuild **once per commit**, not per
  `metaSet`, or a large per-scope KV with per-entry commits degrades to O(n²).
- **Bounds-safety for the new structures** — `scope_table_root`, every `kv_root`, internal
  child pgnos, and overflow pgnos must be range-checked, with recursive validate + per-page
  CRC, mirroring the v4.0 core; a hostile file must yield `Corruption`, never OOB/panic.
- **Upgrade path** — a v4.1 writer upgrades a v4.0 file in place on first metadata write
  (sets `version_minor=1` + `scope_table_root`); a no-metadata file stays byte-compatible
  v4.0. Test both directions.
- Scope-table entry size (~275 B fixed) × many scopes — measure storage for large
  scope-counts (geo); `name` waste on empty names is accepted (revisit only if it bites).

### Sensitive data handling plan

None — generic engine; no secrets, no FireHOL infra details. (Note: callers may *store*
secrets in KV; that is the caller's policy and out of scope for the engine.)

### Implementation plan (ordered)

1. Meta: add `scope_table_root`, bump `meta_size`/`version_minor`; forward-compat
   (reader-skips / writer-refuses) per §5.1.
2. Scope table (fixed-record B+tree keyed by `scope_id`) + per-scope header get/set
   (seek) + `scopeDefine`/`scopeDrop`/`scopeList`/`scopeName`.
3. `version` helpers (`scopeBumpVersion` / `scopeSetVersion`).
4. Per-scope KV **bulk-loaded B+tree** + overflow pages (page/entry formats §D; buffer in
   memory → bulk-rebuild at commit; `O(log n)` reads; `metaGet` missing → `Ok(None)`;
   `TREE_HEIGHT_MAX`; overflow read-by-count) + `metaSet`/`metaGet`/`metaDelete`/`metaList`;
   `FILE` (scope 0) target; error mapping §C.7.
5. Extend the derived allocator to walk `scope_table_root` + `kv_root`s + KV runs.
6. Shared Rust+Go conformance — **cross-read behavioral** (each reads the other's metadata
   identically; shape may differ), per-language goldens, page/entry encoding per spec §D.

### Validation plan

- Conformance: **cross-read behavioral** — each impl reads the other's metadata identically
  (tree shape may differ); the page/entry encoding (spec §D) is the byte-level contract.
  Per-language goldens.
- Forward-compat: a v4.0 reader reads a v4.1 file's IP tree correctly and ignores metadata;
  a v4.0 writer refuses a v4.1 file; a v4.1 writer upgrades a v4.0 file on first metadata
  write.
- Validation: malformed key (NUL/empty/>1024), bad `type==0` value (invalid UTF-8 over the
  whole reassembled value) → typed errors; **`TREE_HEIGHT_MAX` and overflow cycle/length
  defenses**; fuzz the scope-table + KV reader (never OOB/panic/loop).
- CoW: full-rewrite KV per-scope per-commit; allocator reclaims old KV runs;
  crash-recovery (double-meta) with metadata present.
- Real-use: round-trip a realistic feed metadata set (name/version/type + KV attrs).

### Artifact impact plan

- `design-iprange-v4-scope-api.md` — already updated (§C). `AGENTS.md` (iprange) — add
  v4.1/metadata build+test commands when code lands. The locked `design-iprange-v4-livedb.md`

### Open decisions

All resolved (see §C and Decisions above). None blocking. Remaining tunables are
implementation-defined (M-class): scope-table and KV-tree fanout, and the inline-vs-overflow
value threshold (which **must be fixed** so Rust and Go produce identical trees — §C.4).
`name` is **fixed-256** (true seek); per-scope KV is **unbounded** (bulk-loaded B+tree,
no cap).
