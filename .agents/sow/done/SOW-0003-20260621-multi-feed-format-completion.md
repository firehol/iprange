# SOW-0003 - Complete the binary format: interleaved multi-feed merged file

## Status

Status: completed

Sub-state: Delivered. v3.1 spec LOCKED (6 external review rounds → round 6 zero
P0/P1/P2 across all five reviewers, 21 fixes). Rust + Go merge constructor + catalog
reader implemented and **byte-identical** (10 new multi-feed conformance cases,

## Requirements

### Purpose

Finish the **on-disk binary format completely — including the interleaved
multi-feed merged file — before building the higher-level processing engine**, so
the three language implementations and the conformance corpus are pinned to a
frozen, complete contract. Doing multi-feed after the engine would force a
multi-front refactor (spec + Rust + Go + future C + corpus + SDK consumers);
doing it now, additively on the already-extensible v3 container, avoids that
churn.

Fit-for-purpose: the first consumer is `update-ipsets`, which needs **multi-x-multi**
capability — for every IP, the *set* of feeds containing it — to serve its
"which lists contain an IP" query and its N×N overlap matrix from one interleaved
structure instead of O(N²) pairwise scans. The portable interleaved file is also
the right primitive for the Netdata netflow/topology SDK (one mmap lookup →
"which threat feeds is this flow IP in").

### Assistant Understanding

Facts (verified this session, file:line):

- **No interleaved multi-feed exists anywhere today.** The current Go
  `pkg/iprange` (firehol/update-ipsets @ 7e0312c322dd) does **batched pairwise
  overlap *counting***, not a single interleaved membership sweep:
  - N×N comparison is an explicit O(N²) pair loop
    (`pkg/engine/output_comparison.go:209`), pruned by disjoint/identical
    fast-paths + a persistent ledger cache (`:90`), survivors delegated to
    `iprange.CompareSourcePairs` (`:350`), which groups by-left and runs a
    one-to-many heap sweep (`pkg/iprange/compare_pairs.go:350`). Output is
    **counts only** — `CommonIPs`, `CombinedIPs` (`pkg/iprange/set_ops.go:262`,
    `pkg/engine/types.go:70`).
  - Merges (level1–4) = N-ary union then subtract exclusions
    (`pkg/engine/feed_body_stage.go:473,487`); the union is **anonymous** —
    provenance is discarded.
  - The k-way union heap carries a `sourceID` per range internally
    (`pkg/iprange/iter6_ops_indexed.go:34`) but it is **ephemeral**, dropped
    after the merge — the tag-with-source machinery is almost there in memory,
    never persisted.
  - "Which feeds contain IP X" is a shipped feature
    (`cmd/update-ipsets/query.go`, `pkg/engine/query.go:20`) computed by
    **scanning every feed independently** — no membership index.
- **The v3 format already anticipates this.** SOW-0001 locked **O3 = explicit
  `{start,end,value_id}`** and **O4 = interned sorted id-lists for membership
  sets**; `binary-format-v3.md` §10 already defines `type_id == 1` as a
  membership set (≥1 strictly-ascending LE u32 feed-ids). SOW-0001's downstream
  item #4 is exactly "merged multi-feed format (catalog + merged index +
  feed-id registry)". This SOW pulls item #4 ahead of Step 2 (engine).
- The production update-ipsets daemon runs on its own private native-endian
  format; completing our portable format **blocks nothing operational**.

Inferences:

- "Interleaved is crucial" decomposes into **two separable deliverables**: the
  interleaved *on-disk format* (this SOW) and the interleaved *in-memory
  algorithm* that produces it efficiently and derives the N×N matrix in one pass
  (Step 2 engine). The format only needs the former; the producer here can be a
  correctness-first reference writer.
- Because the v3 container is sectioned + skip-unknown + version_minor-additive,
  multi-feed is a **superset** of v3.0 single-feed (new catalog section + reuse
  of the existing index/membership machinery), not a redesign.

Unknowns (resolve during spec drafting, not blocking):

- Exact catalog identity field set (start minimal, mirroring `feed-meta` §7).
- Whether the merged index reuses the existing `index` kind (with membership
  `value_id`s) verbatim or needs a flag — expected: same kind, membership values
  already covered by §10 type_id=1.

## Implications And Decisions

Decisions (Costa, 2026-06-21 — accepted recommendations):

1. **Sequencing (1a):** complete the multi-feed format as its own deliverable
   (this SOW), **before** the Step 2 engine. Reason: freeze the contract once;
   avoid the multi-front refactor. Classification: long-term-best.
2. **Catalog (2a):** the feed-id → identity catalog lives **in the file**
   (self-describing, portable). Reason: a portable signed format whose meaning
   lives outside the file is not portable; the Netdata SDK consumes a standalone
   artifact.
3. **Feed-id assignment (3a):** **caller-supplied stable registry** — ids fixed
   once a feed exists and survive add/remove. Reason: update-ipsets tracks
   retention/history; sorted-name renumbering would break historical
   comparability of merged files.
4. **Lookup accelerator (4a):** ship the interval map with **binary search**
   (O(log M)) now; keep the kind-6 accelerator **reserved**; add it later
   additively only if full-scale profiling demands. Reason: accelerators are
   skippable sections — zero-churn to add; don't pre-optimize without evidence.
5. **Merge rule (stated, not a fork):** overlapping input feeds split into
   **elementary intervals**; coalesce adjacent intervals **only when their
   feed-set is identical**. This is the unique deterministic, byte-identical rule.
6. **S2 — merge algorithm location (c):** the **format library** owns the
   low-level merged writer/reader (bytes + validation) **plus a thin, deterministic
   k-way feed-merge helper** (the interleaved-file constructor). General set-algebra
   (intersect/exclude/diff over arbitrary values) stays in the Step-2 engine. Reason:
   the feed-merge is fully determined (the feed-set *is* the overlap result, no
   value-conflict resolution), so it is the format's constructor, not the engine's
   general overlap resolution; this delivers real multi-x-multi from "the format."

Additive design shape (derived from the locked decisions; confirmed low-churn —
no single-feed bytes change):

- Multi-feed = **`version_minor = 1` (v3.1)**, `header_size` stays 72 (no new header
  fields). A v3.0 reader reads a merged file as an ordinary index whose values are
  opaque `type_id == 1` membership sets (skips the `must_understand = 0` catalog) — a
  valid degraded read.
- **Catalog = kind 4** (repurposing the reserved-never-emitted slot; §8/§4 already
  pin its position), align 8, one §7-style identity block per feed, with an
  **explicit stored `feed_id`** per entry (decision 3a — stable, survives add/remove).
- **Merged index = the existing kind-2 index**; every record's `value_id` → a
  `type_id == 1` membership set (already defined/validated in §10).
- **Merge rule = the existing §9 coalescing** ("merge adjacent iff values
  byte-identical AND contiguous" = "same feed-set").

Scope boundary recorded: this SOW delivers the **format** (layout + read + the
deterministic k-way merge constructor with byte-identity). The **optimized multi-way
producer** (the O(total) sweep that also derives the N×N matrix in one pass) is
**Step 2 engine**, not this SOW.

## Outcome

The interleaved multi-feed merged file is delivered as **v3.1** — additive over the
locked v3.0 single-feed format, with **zero v3.0 on-disk byte changes**. Both the Rust
reference and the pure-Go port implement the catalog section (kind 4), the deterministic
k-way merge constructor (§13.3), membership lookup (§13.4), and the full §13.5/§15
validation, and produce **byte-identical** files verified across 21 conformance cases
(the cross-language guard). This is the "which feeds contain this IP in one lookup"
primitive update-ipsets needs for multi-x-multi and the Netdata SDK consumes. The format
is now complete enough to build the Step-2 engine on a frozen contract — the original
goal of doing the format first to avoid multi-front refactoring.

## Lessons Extracted

- **Reuse the canonicalizer for byte-identity.** Feeding the merge sweep's elementary
  intervals back through the *existing* single-feed `Writer` (sort/coalesce/intern/
  assemble) meant byte-identity, coalescing, and interning came for free and the
  single-feed golden files guarded the `build_inner` refactor — no parallel encoder to
  keep in sync.
- **Count-based active set beats a set for sweep correctness.** A `feed_id → count`
  multiset makes the active set independent of intra-address event order, so a feed that
  ends at `b-1` and resumes at `b` (contiguity) nets to "still active" regardless of
  whether adds or removes are applied first — exactly the §13.3 guarantee.
- **A locked spec still spawns small implementation decisions.** `version_minor == 1`
  becoming "merged" silently invalidated two forward-compat tests (they had used minor 1
  as a stand-in for "future"); the same-failure scan caught both. Defining a version is a
  breaking reinterpretation for anything that used it as a placeholder.
- **rustfmt is version-sensitive.** The repo was not clean under the local rustfmt 1.8.0
  and CI does not enforce fmt; per Costa, the resolution was to format the whole tree
  (Rust + Go) and commit, normalizing on the current toolchain.
